This execution plan covers:

- internal distribution of the current package;
- reviewer-role assignment;
- review sequence;

This execution plan does not authorize:

## Review Roles

The package owner should aim for two to four internal reviewers with
complementary roles.

### 1. Architecture and governance reviewer

Primary focus:

- contribution clarity;
- architecture-level coherence;
- constitutional governance framing;
- bounded local governance domains;
- functional service modules;
- contestability framing.

### 2. Evidence and citation reviewer

Primary focus:

- evidence-boundary compliance;
- claim discipline;
- citation-to-proposition fit;
- bibliography-readiness issues;
- residual sensitive-claim language.

### 3. Human--AI and evaluation reviewer

Primary focus:

- bounded AI-assistance framing;
- evaluation-plan credibility;
- reviewability and contestability;
- limitations and future-work discipline.

### 4. Consistency and editorial reviewer

Optional but useful when available.

Primary focus:

- Markdown/LaTeX equivalence;
- terminology coherence;
- readiness/status wording;
- clarity and non-redundancy.

One reviewer may cover more than one role if the reviewer’s expertise is
adequately broad, but each role must be explicitly assigned.

## Review Criteria

All reviewers should apply the following criteria:

- the manuscript must remain conceptual and prototype-stage;
- no operational, production, adoption, financial, or tokenomics claims may
  be inferred or normalized;
- no section should imply active governance bodies, deployed institutional
  authority, or implementation inventory;
- Markdown and LaTeX must remain substantively synchronized;
- citations must support the propositions for which they are used; and
- readiness language must remain internal-only and non-public.

## Review Order

The package owner should run the review in the following order.

### Step 1. Package-owner preflight

Before sending the package:

- confirm the version-lock hashes;
- complete the author pre-distribution checklist in
  `v0.5-internal-review-package.md`;
- confirm that the package contents match the package manifest;
- confirm that all external-use blockers remain active.

### Step 2. Primary architecture pass

The architecture and governance reviewer should review first. This establishes
whether the current `v0.5` line is coherent enough to justify the rest of the
review cycle.

### Step 3. Evidence and citation pass

The evidence and citation reviewer should review second, using the
architecture pass as context but remaining independent in judgment.

### Step 4. Human--AI and evaluation pass

The human--AI and evaluation reviewer should review third, focusing on
whether the current RQ framing, AI-assistance limits, and contestability
discussion remain properly bounded.

### Step 5. Consistency/editorial pass

If assigned, the consistency reviewer should review after the substantive
passes. This pass should not override substantive findings but should identify
Markdown/LaTeX drift, terminology inconsistency, or readiness-label friction.

### Step 6. Consolidation

The package owner should consolidate all findings only after the individual
forms are received.

Reviewers should not edit each other’s forms or see each other’s full review
before consolidation unless explicitly approved.

## Confidential Package Handling

The package owner should:

- assign each package to a named internal reviewer;
- record reviewer identifier, package date, and file hashes supplied;
- use a private channel for package delivery and return;
- avoid public links, public trackers, and unrestricted shared drives;
- avoid broadcasting reviewer identities to other reviewers by default;
- provide only the smallest additional context requested; and
- set a return or secure-deletion expectation for package copies.

The package remains confidential internal material. Opening the package does
not change the external blockers.

## How to Record Findings

Every reviewer must use the role-specific form assigned by the package owner.

The package owner should require:

- one form per reviewer;
- one finding block per distinct issue;
- section references for every substantive finding;
- explicit severity assignment;
- explicit statement of whether the issue blocks the next readiness
  assessment; and
- `SENSITIVE — discuss privately` handling where necessary.

## Severity and Decision Handling

The package owner should normalize findings into the following decision path:

### Critical

Use for:

- evidence-boundary violations;
- sensitive disclosure;
- materially unsupported central claims;
- citation misuse affecting a central proposition;
- Markdown/LaTeX inconsistency that changes meaning.

Default handling:

- blocks next readiness assessment;
- requires correction before any new internal-review conclusion is accepted.

### Major

Use for:

- architecture ambiguity affecting the central contribution;
- evaluation framing that materially changes interpretation;
- terminology that materially changes scope or maturity reading;
- readiness wording that could be misread as external authorization.

Default handling:

- usually blocks next readiness assessment until corrected or explicitly
  accepted with rationale.

### Minor

Use for:

- bounded clarity issues;
- local wording problems;
- non-central citation or terminology cleanup;
- non-substantive redundancy.

Default handling:

- does not automatically block the next readiness assessment.

### Editorial

Use for:

- copyediting;
- formatting;
- stylistic consistency.

Default handling:

- does not block readiness.

## Consolidation and Decision Record

After all assigned reviews are returned, the package owner should create a
single consolidation record in a later phase. That record should:

- list all findings by reviewer-neutral identifier;
- classify severity;
- assign a disposition:
  - accepted;
  - partially accepted;
  - rejected;
  - deferred;
- identify whether the issue requires:
  - manuscript patch;
  - readiness cleanup;
  - bibliography cleanup;
  - boundary clarification; or
  - no action;
- record any author decisions required; and
- recommend the next gate:
  - major internal revision;
  - minor internal revision;
  - readiness reassessment; or
  - no advancement.

## Decision Logging Rules

The package owner should keep the following distinct:

- reviewer findings;
- package-owner consolidation;
- author decisions; and
- readiness status changes.

A favorable review outcome does not automatically:

- open controlled external review;
- open publication;
- authorize submission; or
- authorize external circulation.

Only an explicit later readiness gate may consider any of those actions.

## Minimum Completion Criteria

The internal review execution should not be considered complete until:

- all assigned reviewers return their forms or are formally marked unable to
  assess;
- the package owner verifies version-lock consistency across all distributed
  copies;
- findings are consolidated into a single decision record; and
- readiness status is updated in a later explicit phase.

## Post-Review Transition Options

After consolidation, the likely next step should be one of:

- targeted internal revision;
- readiness cleanup;
- bibliography cleanup;
- a combined minor patch and readiness reassessment; or
- package closure without advancement.

This plan does not choose that next step in advance.

## Forbidden Actions in This Phase

This execution-planning phase must not:

- edit the manuscript;
- edit `references.bib`;
- authorize controlled external review;
- authorize publication or submission;
- authorize external circulation; or
- weaken the confidentiality or evidence-boundary requirements.
