# Claims Proposal

Publication: `PUB-ADS-001`

Version: `ADS-001-v0.6-draft`

This file proposes claim-register handling for ADS-001. It is not itself a
second claims register and does not authorize central-register updates.

## Existing mappings

| ADS-001 area | Existing claim IDs | Treatment |
|---|---|---|
| Conceptual federated architecture | `AX-024`, `AX-026`, `AX-030` | Use approved bounded design-proposal wording. |
| Maturity boundary | `AX-003`, `AX-004`, `AX-019` | Use repository taxonomy wording; do not infer levels. |
| Functional service modules | `AX-018`, `AX-030` | Treat classes and failure behavior as conceptual abstractions. |
| Human-supervised AI assistance | `AX-014` | Treat assistance as bounded and advisory. |
| Risk boundaries and contestability | `AX-012`, `AX-020`, `AX-022` | Treat enforcement and effects as unvalidated proposals. |
| Evaluation measures | `AX-023` | Treat all measures as future criteria, not results. |

## Candidate unique claims

The following are candidates for new central-register rows only if a duplicate
search confirms that no existing row covers them:

- The Technical Authorization and Institutional Justification pairing is a
  proposed ADS-001 invariant, not an implemented gate.
- The seven-field traceability schema is a conceptual record requirement, not
  an implemented provenance system or integrity guarantee.
- Module failure behaviors are proposed design rules, not guaranteed system
  defaults.
- Reputational Collateral and Sampling for Secondary Review are unresolved
  friction proposals requiring legal, ethical, and empirical review.

The central register must remain unchanged if these candidates are already
covered by existing claims. Any accepted new row requires a stable `AX-###`
identifier, classification, allowed wording, evidence state, and status.
